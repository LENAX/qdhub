## 2. 修复 1：Container 启动时执行完整迁移

### 2.1 问题

- `container.runMigrations()` 仅执行：`001_init_schema`、`002_auth_schema.*`、`006_seed_default_admin.*`。
- 未执行：`002_seed_mapping_rules`、`003_sync_plan_migration`、`004_api_sync_strategy`、`005_sync_plan_default_params`、`007_daily_adj_factor_trade_date_expand`。
- 全新部署时缺少 `sync_plan`、`sync_task`、`sync_execution`、`api_sync_strategies` 等表，同步计划与 API 同步策略功能会报错。

### 2.2 方案

- 在 `runMigrations()` 中，在现有 001、002_auth、006_seed 基础上，**按顺序**执行以下迁移文件：
  - `002_seed_mapping_rules.up.sql`（仅 SQLite：含 `INSERT OR IGNORE`）
  - `003_sync_plan_migration.up.sql`（标准 SQL）
  - `004_api_sync_strategy.up.sql`（仅 SQLite：含 `INSERT OR REPLACE`、`randomblob()`）
  - `005_sync_plan_default_params.up.sql`（标准 SQL）
  - `007_daily_adj_factor_trade_date_expand.up.sql`（标准 SQL）
- 执行顺序：001 → 002_auth → **002_seed_mapping_rules** → **003** → **004** → **005** → 006_seed → **007**。
- 当 driver 非 SQLite 时，跳过 002_seed_mapping_rules 与 004（或后续提供 Postgres/MySQL 版本后再纳入）。

### 2.3 涉及文件

- `qdhub/internal/infrastructure/container/container.go`：扩展 `runMigrations()`，增加 `runOptionalMigrationFile(path)` 与按序调用。

### 2.4 验收

- 全新 SQLite 库 + 仅执行 `./bin/qdhub server` 后，存在表：`sync_plan`、`sync_task`、`sync_execution`、`api_sync_strategies`，且可创建同步计划、解析依赖、触发执行。
- 已验收：删除 `./data/qdhub.db` 后启动 server，迁移按序执行，上述表均存在。

### 2.5 非 SQLite 驱动说明

- 当 `database.driver` 为 `postgres` 或 `mysql` 时，`002_seed_mapping_rules` 与 `004_api_sync_strategy` 会跳过（日志：Skipping ... SQLite only），其余迁移（003、005、007）会执行。
- 使用 Postgres/MySQL 时若需同步策略等数据，建议部署时先执行 `./bin/qdhub migrate up` 或 `scripts/migrate.sh up`（脚本支持多库语法转换），再启动 server。
